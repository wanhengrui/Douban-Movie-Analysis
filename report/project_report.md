# 基于豆瓣Top250电影数据的采集、分析与推荐系统

> 项目报告 | 2026年6月

---

## 一、项目概述

本项目以豆瓣电影Top250为数据源，构建了一个完整的数据采集、清洗、分析、推荐与可视化系统。用户可通过Web界面查询多维度的电影统计信息、获取基于剧情内容的智能推荐，并与AI对话探索数据。

**核心数据**：250部电影，10个字段（名称、评分、年份、国家、类型、导演、演员、片长、剧情简介、详情页URL）。

**评分范围**：8.4~9.7，均分8.95。**年代跨度**：1931~2023。

## 二、系统架构

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  数据采集层   │ →  │  数据处理层   │ →  │  应用层       │
│              │    │              │    │              │
│ 列表页爬虫    │    │ 缺失值填充    │    │ Web界面      │
│ 详情页爬虫    │    │ 字段标准化    │    │ AI问答       │
│ 补爬脚本      │    │ 解析错位修复  │    │ TF-IDF推荐   │
│              │    │              │    │ 分析报告      │
└──────────────┘    └──────────────┘    └──────────────┘
       ↓                   ↓                   ↓
  movies.csv        movies_cleaned.csv     Flask + HTML/CSS/JS
```

**技术栈**：Python 3.14、Requests + BeautifulSoup（爬虫）、Pandas + NumPy（数据处理）、Scikit-learn（TF-IDF推荐）、DeepSeek API（AI分析）、Flask（Web后端）、原生HTML/CSS/JS（前端）。

## 三、功能模块

### 3.1 数据采集

采用双策略爬取方案：

| 策略 | 目标页面 | 获取字段 | 说明 |
|------|----------|----------|------|
| 桌面版列表页 | movie.douban.com/top250 | title, score, year, country, genre, director, actors, detail_url | 每页25部，共10页 |
| 手机版详情页 | m.douban.com/movie/subject/{id} | runtime, summary | 逐部爬取 |

导演和主演信息隐藏在列表页 `<p>` 标签的第一行（格式："导演: XXX   主演: XXX"），原版爬虫丢弃了这一行，本系统通过字符串分割将其提取出来，避免了对250个详情页的额外请求。

### 3.2 数据处理

数据清洗模块处理了三种类型的异常：

1. **日期串位**（3部）：部分国产老电影的年份字段含重映日期，如"1961 / 1964(中国大陆上映)"，按"/"分割后导致country和genre错位。
2. **导演/演员缺失**（15部）：某些电影的`<p>`标签结构不同，"导演:"或"主演:"标签缺失，正则解析失败。
3. **缺失值**：runtime 1部、summary 3部为空，用"未知"填充。

### 3.3 TF-IDF 电影推荐

将250部电影的剧情简介文本做TF-IDF向量化：

- **TF（词频）**：一个词在单部电影简介中出现的频率
- **IDF（逆文档频率）**：一个词在所有电影简介中出现的稀有程度

两者相乘得到每部电影的文本特征向量，再通过余弦相似度计算任意两部电影的内容相似度。用户输入电影名后，系统返回内容最接近的Top5电影。

### 3.4 AI 集成

系统通过 DeepSeek API（deepseek-v4-flash 模型）实现三项AI功能：

1. **智能问答**：用户自由提问，AI基于实时计算的数据统计和电影详情回答
2. **推荐解释**：TF-IDF返回相似电影后，AI基于剧情简介文本解释推荐理由
3. **报告生成**：自动计算6个维度统计摘要，交由AI撰写专业分析报告

### 3.5 可视化

生成三张图表用于项目展示：
- 评分分布直方图
- 国家/地区分布横向条形图
- 国家平均评分排名图

## 四、重点技术细节

### 4.1 合拍片处理

多国合拍片（如"美国 / 英国 / 加拿大"）在统计时需拆分计算。使用 `explode()` 方法将一行拆为多行，每行对应一个国家，各自计入统计。这样"星际穿越"会同时贡献给美国、英国、加拿大的统计。

### 4.2 TF-IDF 参数选择

`max_features=3000` 将中文词汇空间限制在3000维。对于250篇简介文本，这个维度在计算效率和语义表达之间取得了平衡。过低的维度会丢失区分信息，过高则计算量大且可能引入噪声。

### 4.3 异步AI调用

Web端AI问答使用异步模式：前端发起请求后不阻塞界面，显示"思考中"占位；后端在新线程中调用API，完成后通过回调更新界面。确保用户体验流畅。

### 4.4 数据清洗中的硬编码修复

15部电影的导演/演员数据通过硬编码字典补全。这15部电影在列表页的HTML结构与其余235部不同，属于豆瓣页面的个案格式异常。对于250条数据中的18条边缘情况（7.2%），硬编码修复比写通用正则更安全、更可解释。

## 五、遇到的困难与解决方案

### 困难1：列表页逐步收紧反爬策略

**现象**：开发过程中，列表页从正常可访问 → 间歇性403 → 302跳转至登录页。

**原因**：豆瓣对未登录用户的访问限制日趋严格，高频请求后IP会被临时标记，需要登录态才能继续。

**解决方案**：
1. 在请求头中加入Cookie字段（从登录浏览器复制），使爬虫以登录用户身份访问
2. 设计断点续传机制：`fix_data.py` 可读取已有CSV，仅对缺失字段重新爬取，避免全量重爬
3. 增加随机延迟和User-Agent池切换，降低请求频率

---

### 困难2：部分电影解析失败（15部导演/演员缺失）

**现象**：运行detail_spider.py后，有15部电影的director和actors字段为空。

**分析**：列表页`<p>`标签结构为"导演: XXX   主演: XXX"。解析代码通过检测这两个标签来提取内容。但部分电影的`<p>`标签缺少这些标签——可能是HTML结构与常规不同（如动画电影、外语片等），导致条件判断失败。

**解决方案**：在数据清洗模块中添加硬编码修复字典。这15部电影中包括黑客帝国三部曲、部分动画电影等。由于无法实时访问豆瓣页面调试（IP被封），采用已知数据补全是最可靠的方式。

---

### 困难3：手机详情页限流导致数据缺失

**现象**：爬取到第230部左右时，多个连续请求返回空内容。

**原因**：连续250次请求触发了豆瓣手机端的频率限制。

**解决方案**：
1. 设置随机延迟（1.5~2.5秒）+ 每50部休息10秒
2. 检测返回页面长度（<5000字节为反爬页面），自动重试
3. 编写独立补爬脚本 `fix_data.py`，仅对缺失项重新爬取，最终缺失数从12部降至1部

---

### 困难4：AI 回答质量依赖 Prompt 设计

**现象**：最初AI回答过于笼统，缺乏数据支撑。例如问"推荐几部电影"，AI泛泛而谈，没有引用具体电影名或评分。

**原因**：DeepSeek API本身不了解本项目的250部电影数据集。如果仅发送用户问题，AI只能凭训练数据猜测，无法给出基于真实数据的回答。

**解决方案**：
1. 在每次对话的system prompt中注入实时计算的数据摘要——国家Top5、类型Top5、高分电影列表等，使AI"知道"数据集内容
2. 对于推荐类问题，先用Python执行TF-IDF匹配，将Top5结果+剧情摘要嵌入用户消息，AI看到具体数据后再生成自然语言解释
3. 检测用户问题中的关键词（"推荐""类似""相似"），自动触发TF-IDF预处理流程
4. 最终AI能引用具体电影名、评分、剧情关键词来回答，而非空泛描述

---

### 困难5：AI 调用的异步处理

**现象**：点击"发送"或"推荐"按钮后，Web界面卡顿10~30秒，无法进行任何操作。

**原因**：DeepSeek API调用耗时较长（网络延迟 + 模型推理），而Flask默认同步处理请求。前端JavaScript的fetch调用也是同步等待响应，导致UI线程被阻塞。

**解决方案**：
1. Flask后端路由中，API调用放在独立线程中执行（`threading.Thread`），不阻塞Flask主线程
2. 前端使用异步fetch + Promise链：发送请求后立即显示"AI思考中..."占位消息，响应返回后替换为实际内容
3. 输入框在等待期间禁用（`disabled=true`），防止重复发送

---

### 困难6：前端展示形态的迭代

**现象**：项目经历了三次界面重构——最初是命令行文本菜单，后来改为tkinter桌面窗口，最终定型为Flask Web应用。

**原因**：
1. 命令行菜单太简陋，无法展示推荐卡片和对话气泡
2. tkinter窗口样式难以美化，且不便于答辩演示（需Python环境）
3. Web方案浏览器即可打开，CSS灵活度高，更适合展示

**解决方案**：选择Flask单文件方案——后端路由和HTML模板写在同一个`.py`文件中，无需前端构建工具。界面采用卡片式布局（推荐结果用圆角卡片+评分标签、对话用气泡消息），所有样式内嵌在`<style>`标签中。最终代码零前端依赖，`pip install flask` 即可运行。

## 六、项目总结

本项目从零构建了一个完整的数据分析系统，涵盖了数据采集、清洗、分析、推荐和展示的全流程。在技术选型上，爬虫层采用双策略规避反爬，分析层引入TF-IDF算法和AI大模型提升智能化程度，展示层通过Web界面提供友好的交互体验。

通过这个项目，实践了以下能力：
1. Web爬虫设计与反爬对策
2. 数据清洗的工程化方法
3. 文本向量化与相似度计算的原理和实现
4. 大语言模型API的集成与Prompt Engineering
5. 全栈Web应用的构建（Python后端 + 原生前端）
6. Git版本控制与协作规范

**后续改进方向**：引入更多数据源（如IMDb评分对比）、优化推荐算法（融合导演/演员/类型等结构化特征）、增加用户个性化推荐功能。
